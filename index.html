What's Fixed Here Issues not in the official docs
▶
🔴 RTX 5090 (sm_120 / Blackwell) crashes with stable PyTorch Stable PyTorch (2.2.0+cu121) does not include Blackwell kernels. Any GPU op fails with RuntimeError: CUDA error: no kernel image is available for execution on the device. Fix: use PyTorch nightly cu128 with TORCH_CUDA_ARCH_LIST="12.0".
🔴 Old PyTorch silently served from Docker layer cache Even with the correct Dockerfile, Docker may serve a cached layer with the wrong PyTorch version. Fix: delete the old image with docker rmi and rebuild with --no-cache.
🟡 Model weights (~14 GB) re-downloaded on every container run Without a persistent cache mount, weights download fresh each time. Fix: mount ./hf_cache and set HF_HOME=/workspace/hf_cache.
🟡 Flash Attention 2 warning about GPU init order Load the model on CPU first with low_cpu_mem_usage=True, then call .to('cuda:0'). This is shown in the Load Model step.
ℹ Verified working results on RTX 5090 PyTorch 2.12.0.dev+cu128 · CUDA 12.8 · 7.54B parameters · 14.09 GB VRAM used · 15.81 GB free

Prerequisites Docker · NVIDIA Container Toolkit · HuggingFace token
▶
$bash — host machine
# Install Docker
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER && newgrp docker

# Install NVIDIA Container Toolkit
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
  sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg

curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

sudo apt-get update && sudo apt-get install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker

# Verify GPU is visible to Docker
docker run --rm --gpus all nvidia/cuda:12.8.0-base-ubuntu22.04 nvidia-smi
HuggingFace token Get your token at huggingface.co/settings/tokens with read access. The openvla-7b model is public — no access request needed.

Folder Structure Clone repo · create data folders
▶

Clone the OpenVLA repo. Docker files and data folders all live inside it — this is your permanent working directory.

$bash
# Clone the repo
git clone https://github.com/openvla/openvla.git
cd openvla

# Create persistent data folders
mkdir -p hf_cache datasets checkpoints outputs
Final structure inside openvla/ prismatic/ · vla-scripts/ · experiments/ · hf_cache/ · datasets/ · checkpoints/ · outputs/ · Dockerfile · docker-compose.yml · .env
Live code editing The entire openvla/ folder is mounted into the container as a volume. Edit any file on your host machine and changes appear instantly inside the container — no rebuilds needed.
03
Dockerfile nvidia/cuda:12.8.0-devel · PyTorch nightly cu128
▶
⚠ Use the devel variant of the CUDA base image The runtime variant lacks compiler headers — Flash Attention build will fail. Must be cudnn-devel.
$Dockerfile
FROM nvidia/cuda:12.8.0-cudnn-devel-ubuntu22.04

ENV DEBIAN_FRONTEND=noninteractive
ENV PYTHONUNBUFFERED=1
# Required for RTX 5090 / Blackwell (sm_120)
ENV TORCH_CUDA_ARCH_LIST="12.0"

# System dependencies
RUN apt-get update && apt-get install -y \
    python3.10 \
    python3.10-dev \
    python3-pip \
    git \
    git-lfs \
    ninja-build \
    build-essential \
    wget \
    curl \
    && rm -rf /var/lib/apt/lists/*

# Make python3.10 the default
RUN update-alternatives --install /usr/bin/python python /usr/bin/python3.10 1 && \
    update-alternatives --install /usr/bin/pip pip /usr/bin/pip3 1

RUN pip install --upgrade pip setuptools wheel

# PyTorch nightly cu128 — REQUIRED for RTX 5090 (Blackwell / sm_120)
# Stable PyTorch does NOT support sm_120 and crashes at runtime
RUN pip install --pre --upgrade \
    torch \
    torchvision \
    torchaudio \
    --index-url https://download.pytorch.org/whl/nightly/cu128

# OpenVLA Python dependencies (pinned versions from official repo)
RUN pip install \
    "transformers==4.40.1" \
    "tokenizers==0.19.1" \
    "timm==0.9.10" \
    "huggingface_hub" \
    "accelerate" \
    "peft" \
    "draccus" \
    "einops" \
    "pillow" \
    "wandb" \
    "tensorflow==2.15.0" \
    "tensorflow-datasets==4.9.3" \
    "dlimp @ git+https://github.com/moojink/dlimp_openvla"

# Flash Attention 2 — compiled for Blackwell
RUN pip install packaging ninja && \
    pip install "flash-attn==2.5.5" --no-build-isolation

WORKDIR /workspace
04
docker-compose.yml Volume mounts · GPU · shm · env
▶
$docker-compose.yml
services:
  openvla:
    build: .
    image: openvla:latest
    runtime: nvidia
    environment:
      - NVIDIA_VISIBLE_DEVICES=all
      - NVIDIA_DRIVER_CAPABILITIES=compute,utility
      - HF_TOKEN=${HF_TOKEN}
      - HUGGING_FACE_HUB_TOKEN=${HF_TOKEN}
      - HF_HOME=/workspace/hf_cache
    volumes:
      - .:/workspace                         # entire openvla folder → container
      - ./hf_cache:/workspace/hf_cache       # model weights — persistent
      - ./datasets:/workspace/datasets       # robot datasets
      - ./checkpoints:/workspace/checkpoints # saved fine-tuned models
      - ./outputs:/workspace/outputs         # logs and results
    shm_size: '32gb'
    ulimits:
      memlock: -1
      stack: 67108864
    stdin_open: true
    tty: true
    command: bash -c "cd /workspace && pip install -e . --quiet && bash"
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]
shm_size is important PyTorch dataloaders use /dev/shm for multiprocessing. Without enough shared memory, fine-tuning workers crash with cryptic errors.

Build the Docker Image ~30–40 min first run · Flash Attention compiles from source
▶
First build takes 30–40 minutes Flash Attention compiles from source (~15 min). PyTorch nightly downloads ~5 GB. All subsequent builds use Docker layer cache and complete in seconds.
$bash — from inside openvla/
# First-time build
docker compose build

# If rebuilding to fix wrong PyTorch version cached from old image
docker rmi openvla:latest
docker compose build --no-cache
Build succeeded when you see: naming to docker.io/library/openvla:latest with exit code 0

Load the Full OpenVLA 7B Model ~14 GB download · cached after first run
▶
First run downloads ~14 GB Weights are saved to ./hf_cache/ on your host and reused on every subsequent run. Never re-downloaded.
$bash
docker compose run --rm openvla python -c "
import torch
from transformers import AutoModelForVision2Seq, AutoProcessor

processor = AutoProcessor.from_pretrained(
    'openvla/openvla-7b', trust_remote_code=True)

vla = AutoModelForVision2Seq.from_pretrained(
    'openvla/openvla-7b',
    attn_implementation='flash_attention_2',
    torch_dtype=torch.bfloat16,
    low_cpu_mem_usage=True,
    trust_remote_code=True
).to('cuda:0')

print('Model loaded!')
print('Parameters:', round(sum(p.numel() for p in vla.parameters()) / 1e9, 2), 'B')
print('GPU memory used:', round(torch.cuda.memory_allocated(0)/1024**3, 2), 'GB')
print('GPU memory free:', round(torch.cuda.mem_get_info(0)[0]/1024**3, 2), 'GB')
"
verified output — RTX 5090

LoRA Fine-tuning Single GPU · 32 GB VRAM · BridgeData V2 example
▶

Download BridgeData V2 dataset first (~124 GB), then launch fine-tuning:

$bash — download dataset
# Inside the container or on host (into datasets/ folder)
cd /workspace/datasets
wget -r -nH --cut-dirs=4 --reject="index.html*" \
  https://rail.eecs.berkeley.edu/datasets/bridge_release/data/tfds/bridge_dataset/
mv bridge_dataset bridge_orig
$bash — LoRA fine-tuning
docker compose run --rm openvla \
  torchrun --standalone --nnodes 1 --nproc-per-node 1 \
  vla-scripts/finetune.py \
  --vla_path              openvla/openvla-7b \
  --data_root_dir         /workspace/datasets \
  --dataset_name          bridge_orig \
  --run_root_dir          /workspace/checkpoints \
  --adapter_tmp_dir       /workspace/outputs/adapter_tmp \
  --lora_rank             32 \
  --batch_size            16 \
  --grad_accumulation_steps 1 \
  --learning_rate         5e-4 \
  --image_aug             True \
  --save_steps            1000
OOM on your GPU? Reduce --batch_size and increase --grad_accumulation_steps proportionally to maintain effective batch size. The RTX 5090 has 32 GB VRAM — --batch_size 16 uses ~72 GB with bfloat16.
Using a custom dataset? Convert to RLDS format, register a dataset config in prismatic/vla/datasets/rlds/oxe/configs.py, and a transform in transforms.py. See the official README for full instructions.

Troubleshooting Known issues and fixes
▶
sm_120 is not compatible + RuntimeError on any CUDA op PyTorch is wrong version. Run docker rmi openvla:latest then docker compose build --no-cache. The old image was served from Docker layer cache.
unknown instruction: cat (did you mean cmd?) A shell command accidentally got written into the Dockerfile. Open with nano Dockerfile, delete all content, and paste only the Dockerfile lines from Step 03.
yaml: mapping values not allowed in this context docker-compose.yml has bad formatting or stray characters. Open with nano docker-compose.yml, delete everything, and paste the YAML from Step 04 cleanly.
Model weights re-downloading every run Confirm HF_HOME=/workspace/hf_cache is in your .env and ./hf_cache:/workspace/hf_cache is in the volumes list in docker-compose.yml.
Flash Attention build fails Ensure the base image is cuda:12.8.0-cudnn-devel-ubuntu22.04. The devel variant is required — the runtime variant lacks compiler headers.
tensorflow-datasets error on RLDS data loading Pin to tensorflow-datasets==4.9.3 — this is already in the Dockerfile above. Newer versions break dataset loading.
TensorFlow cuDNN/cuFFT/cuBLAS registration warnings Harmless — TensorFlow and PyTorch both try to register the same CUDA plugins. OpenVLA uses PyTorch for all compute; TensorFlow is only used for RLDS dataset loading.

Fine-tuning Tips for RTX 5090
→Start with --batch_size 16 and --grad_accumulation_steps 1 — effective batch 16, safe on 32GB VRAM for LoRA
→Use --lora_rank 32 — this is the value used in the official LIBERO fine-tuning experiments with 84.7% success rate
→Collect data at 5–10 Hz. OpenVLA has no action chunking — high frequency data causes the model to get stuck in idle steps at inference time
→Avoid pauses and small actions in demonstrations — the model is sensitive to idle steps and may loop on them during rollout
→Collect ~100 demos minimum for a new task. More diverse initial conditions in demos = better generalisation at test time
→Set --image_aug True during fine-tuning and use center crop at eval — the official LIBERO results use 90% area random crop during training
